## 事前準備: アクション経緯の把握

まず、このタスクの全アクション履歴を確認せよ:

```
tq action list --task {{.Task.ID}}
```

- 各アクションの **result** フィールドから、これまでの経緯・成果・失敗理由を把握する
- 前アクションの結果だけでなく、タスク全体の流れを踏まえて判断する

## 共通ルール
- **重複チェック**: 事前準備で取得したアクション一覧を参照し、同じテンプレートのアクティブなアクション（pending / running / waiting_human）がないか確認する
- **重複エラー**（"blocked: active action already exists"）が返された場合: 前アクションの結果から状況が変わっており再実行が妥当なら強制再作成、そうでなければスキップ
- **選択不可テンプレート**: `classify-gh-notification`, `classify-next-action` はシステム内部用のため選択してはならない

## Phase A: 改善提案の抽出

前アクションの結果に「改善提案」「TODO」「別タスクで実施」「今後の課題」等の記述があれば、各提案を以下の基準で価値判断し、すべて満たすもののみタスク化する。満たさない提案はスキップする。

- タスク管理のオーバーヘッド（作成・追跡・完了）に見合う作業量があるか
- 具体的で独立したアクションにつながるか（1行の軽微な修正や曖昧な提案は不可）
- YAGNI に該当しないか（現時点で必要性が明確か）

`/tq` スキルに従い、プロジェクト {{.Project.Name}} にタスクを作成する。

提案がない、またはすべてフィルタされた場合は何もせず Phase B へ進む。

---

## Phase B: 次アクション判断

### 利用可能なテンプレート
- `implement` — 汎用的な実装タスク
- `self-review` — PR のセルフレビュー
- `respond-review` — レビューコメントへの対応
- `fix-ci` — CI 失敗の修正
- `fix-conflict` — コンフリクト解消
- `merge-pr` — PR マージ

### ルール
1. 結果を読み、このタスクで次に必要なアクションを判断する
2. 共通ルールの重複チェックを行った上で、`/tq` スキルに従いタスク {{.Task.ID}} にアクションを作成する
3. `implement` を作成する場合は meta の instruction に具体的な指示を含める
4. 複数アクションが必要な場合は複数作成してよい
5. 次アクションを作成した場合は Phase C をスキップし、完了へ進む

---

## Phase C: タスク完了判定（Phase B でアクション未作成時のみ）

追加アクションが不要と判断した場合、タスクの目的が達成済みかを判定する。

1. 事前準備で取得した全アクション履歴を踏まえて、タスクのタイトル「{{.Task.Title}}」および URL ({{.Task.URL}}) が示す目的が完了しているか判断する
3. 完了していれば、`/tq` スキルに従いタスク {{.Task.ID}} のステータスを done に更新する
4. 判断できない場合は何もしない

---

完了したら `/tq:done` を実行する。result には以下の判断過程を簡潔に記録すること（3-5行）:

- **Phase A**: 改善提案の有無。抽出した場合はタスクIDを、スキップした場合は理由（提案なし／基準未達等）を記載
- **Phase B**: 次アクション作成の有無。作成した場合はテンプレート名と選定理由を、未作成の場合は理由を記載
- **Phase C**: 実行した場合のみ、タスク完了にしたか否かと判断理由を記載
